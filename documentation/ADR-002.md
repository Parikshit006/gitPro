# ADR-002 — Domain-Oriented Modular Architecture

**Document Type:** Architecture Decision Record
**Date:** July 11, 2026
**Author:** Principal Software Architecture Team
**Classification:** Confidential — Engineering Contract

---

## 1. Status

**Accepted**

---

## 2. Context

As established in ADR-001, GitPro will be built as a Modular Monolith for Version 1. While a monolithic deployment topology maximizes development velocity and minimizes operational overhead for our MVP constraints, it carries a significant architectural risk: the tendency for internal boundaries to erode over time, degrading into a highly coupled "Big Ball of Mud." 

To mitigate this risk and preserve our ability to scale toward an enterprise distributed architecture, the internal structure of the monolith must be strictly segregated into logical domains. GitPro is an Engineering Intelligence Platform; its complexity lies in data ingestion, metric computation, and AI reasoning. Mixing these concerns will make the system unmaintainable.

---

## 3. Why Domain Boundaries Are Necessary

Strict domain boundaries are enforced to guarantee that the logical architecture mimics a distributed system, even while operating in a single process. This provides three critical guarantees:

1. **Independent Evolution:** Changes to how we compute the Bus Factor (Metrics Domain) must not require changes to how we store repository metadata (Repository Domain).
2. **Cognitive Isolation:** Developers can work within a single domain without needing to understand the internal state management of other domains.
3. **Future Extraction (Microservices):** When scale demands that a domain (e.g., Analysis or Intelligence) be moved to dedicated infrastructure, clear boundaries ensure the extraction requires only network layer adjustments, not a rewrite of business logic.

---

## 4. Proposed Domains

The GitPro system is partitioned into the following eight distinct logical domains.

### 4.1. Identity
- **Purpose:** Manages user authentication, session state, and user-level authorization.
- **Responsibilities:** User onboarding, authentication verification, and secure storage of user profiles.
- **What belongs inside:** User entities, session tokens, login workflows.
- **What must NOT belong inside:** Repository access controls (which belong to Integrations/Repository), billing logic.

### 4.2. Repository
- **Purpose:** Acts as the source of truth for the codebase being analyzed.
- **Responsibilities:** Storing repository metadata, tracking the raw commit history, and managing repository selection state.
- **What belongs inside:** Repository records, raw commit logs, file structures, and author metadata.
- **What must NOT belong inside:** Complex statistical metric computations or analysis run state.

### 4.3. Analysis
- **Purpose:** Orchestrates the lifecycle of an analysis run.
- **Responsibilities:** Managing the state machine of an analysis job (queued, ingesting, computing, generating, completed, failed), triggering metric computations, and persisting the final analysis artifact.
- **What belongs inside:** Analysis run records, pipeline orchestration logic, and phase state tracking.
- **What must NOT belong inside:** The actual mathematical algorithms for metrics, or the prompt engineering for AI reports.

### 4.4. Metrics
- **Purpose:** The mathematical and statistical engine of GitPro.
- **Responsibilities:** Computing deterministic engineering metrics (e.g., Hotspots, Bus Factor, Temporal Coupling, Code Churn) from raw repository data.
- **What belongs inside:** Algorithms, scoring thresholds, statistical models, and deterministic data transformation logic.
- **What must NOT belong inside:** External API calls, AI inference logic, or presentation formatting.

### 4.5. Intelligence
- **Purpose:** Transforms raw deterministic metrics into explainable, evidence-backed engineering insights.
- **Responsibilities:** Constructing prompt chains, calculating confidence scores based on metric weightings, ensuring evidence traceability, and generating the structured Repository Health Report.
- **What belongs inside:** Confidence scoring rules, AI report structure definitions, evidence linking logic, and prompt templates.
- **What must NOT belong inside:** Raw commit parsing or metric computation algorithms.

### 4.6. Dashboard
- **Purpose:** The presentation and user interaction layer.
- **Responsibilities:** Translating core domain data into view models, handling API requests from the frontend, and organizing data for visual consumption.
- **What belongs inside:** View models, API controllers/resolvers, data aggregation for UI presentation.
- **What must NOT belong inside:** Business logic, metric algorithms, or database mutation rules.

### 4.7. Integrations
- **Purpose:** Manages all communication across the system boundary to third-party services.
- **Responsibilities:** Interacting with the GitHub API (for cloning and metadata) and AI Providers (for report generation), including rate limiting and network failure handling.
- **What belongs inside:** External API clients, provider configuration handling, and network retry policies.
- **What must NOT belong inside:** Core domain logic or decisions about *what* to do with the external data.

### 4.8. Platform
- **Purpose:** Provides cross-cutting technical capabilities required by all other domains.
- **Responsibilities:** Logging, telemetry, configuration management, standard error handling, and database connection pooling.
- **What belongs inside:** Application configuration, logger definitions, generic utility functions.
- **What must NOT belong inside:** Any feature-specific business logic or domain concepts.

---

## 5. Domain Interaction Rules

To prevent coupling, domains are strictly limited in how they may interact.

- **Dashboard** may depend on **Intelligence**, **Metrics**, **Analysis**, **Repository**, and **Identity** (to serve data to the user).
- **Analysis** may depend on **Integrations** (to trigger ingestion), **Metrics** (to compute scores), and **Intelligence** (to trigger report generation).
- **Intelligence** may depend on **Metrics** (to read evidence data) and **Integrations** (to call external AI models).
- **Metrics** may depend on **Repository** (to read raw commit data).
- **Metrics** must **NEVER** depend on **Dashboard**, **Intelligence**, or **Analysis**. It must remain a pure calculation engine.
- **Repository** must **NEVER** depend on **Metrics** or **Intelligence**.
- **Identity** must remain isolated and independent of repository and analysis domains.
- **Platform** may be used by **every** domain, but must depend on **no** other domain.

---

## 6. Dependency Direction Rules

Dependencies must flow in a **Directed Acyclic Graph (DAG)**. 

1. **Inward Flow:** Dependencies flow from highly volatile, external-facing domains (Dashboard, Integrations) inward toward stable, core business domains (Metrics, Repository).
2. **No Circular Dependencies:** Domain A cannot depend on Domain B if Domain B depends on Domain A. If two domains require bi-directional communication, the shared concept must be extracted into a new, lower-level domain, or communication must occur via decoupled asynchronous events.
3. **Interface Segregation:** When a higher-level domain requires data from a lower-level domain, it must communicate through explicit, stable domain interfaces (contracts), rather than directly accessing the lower domain's internal data structures.

---

## 7. Benefits for Future Migration

By rigorously enforcing these boundaries in the V1 Modular Monolith, we secure the following advantages for future scalability:

- **Seam Identification:** The interaction boundaries define the exact seams where network calls will eventually replace in-memory function calls.
- **Painless Extraction:** If the `Analysis` domain becomes too compute-heavy, it can be lifted into a dedicated service. Because `Metrics` and `Repository` do not depend on it, removing it does not break the core data model.
- **Independent Testing:** Domains can be tested in isolation using mocks at the domain boundary, ensuring high test coverage without requiring the entire monolith to be bootstrapped.

---

## 8. Executive Summary

To ensure GitPro remains maintainable and extensible, the application architecture is divided into eight strict logical domains: Identity, Repository, Analysis, Metrics, Intelligence, Dashboard, Integrations, and Platform. Dependencies flow strictly inward from external interfaces (Dashboard, Integrations) toward core logic (Metrics, Repository), strictly prohibiting circular dependencies. This domain-oriented design guarantees that the Modular Monolith remains highly cohesive and loosely coupled, protecting development velocity today while serving as the blueprint for microservice extraction tomorrow.
