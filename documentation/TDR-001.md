# TDR-001 — Technology Stack Selection

**Document Type:** Technology Decision Record
**Date:** July 11, 2026
**Author:** Principal Software Architecture Team
**Classification:** Confidential — Executive Engineering Review

---

## Executive Overview

This document defines the foundational technology stack for the GitPro MVP. The selections herein optimize for the constraints of a two-developer team and a six-week delivery timeline, without compromising the long-term strategic requirements of a Modular Monolith architecture, enterprise self-hostability, and strict domain boundaries. The system uses the best language for each responsibility. Node.js powers the application layer. Python powers repository mining and engineering analysis. Language choice is driven by ecosystem maturity rather than developer convenience.

---


## Technology Selection Principles

- Choose the best tool for each responsibility.
- Favor mature ecosystems over trendy frameworks.
- Optimize for long-term maintainability.
- Keep the architecture provider-agnostic.
- Every technology must support enterprise evolution.
- Avoid unnecessary complexity during MVP.
- Decisions are based on product needs, not popularity.

---

## 1. Frontend

**Selected Technology:** React, TypeScript, Vite, Tailwind CSS, TanStack Query, React Router, Recharts

- **React & TypeScript:** Core presentation layer ensuring type-safety and robust component architecture.
- **Vite:** High-performance build tool enabling rapid local iteration.
- **Tailwind CSS:** Utility-first styling for accelerated UI development and consistent design tokens.
- **TanStack Query:** Handles asynchronous state management, caching, and synchronization with the backend.
- **React Router:** Manages client-side routing for the Single Page Application.
- **Recharts:** Provides enterprise-grade data visualization for engineering metrics and trend analysis.

---

## 2. Backend

**Selected Technology:** Express.js + TypeScript

- **Purpose:** To act as the core application server, hosting the Modular Monolith, exposing RESTful APIs, and orchestrating the analysis pipelines.
- **Why it was selected:** Express.js prioritizes rapid development using existing team expertise. Clean architecture is enforced through internal engineering discipline and strict module boundaries rather than rigid framework conventions.
- **Alternatives considered:** NestJS, Go, Spring Boot.
- **Trade-offs:** We must manually implement and enforce architectural boundaries, dependency injection, and routing structures that heavier frameworks provide out-of-the-box.
- **Why alternatives were rejected:** While heavier frameworks offer structural guarantees, they introduce learning curves and boilerplate that reduce development velocity. Express allows the team to move at maximum speed during the MVP phase while remaining capable of enterprise scale when paired with TypeScript.

---

## 3. Database

**Selected Technology:** PostgreSQL

- **Purpose:** To serve as the primary persistent data store for user identities, repository metadata, analysis run histories, and generated reports.
- **Why it was selected:** PostgreSQL is the industry standard for enterprise relational databases. Crucially, its robust `JSONB` support allows us to store structured, schema-less metric payloads and AI report structures efficiently while retaining relational integrity for users and repositories.
- **Alternatives considered:** MongoDB, MySQL.
- **Trade-offs:** Schema migrations require rigorous management compared to schema-less databases.
- **Why alternatives were rejected:** GitPro's core domain (Users have Repositories, Repositories have Analysis Runs) is deeply relational. MongoDB forces unnatural data duplication for relational entities. MySQL's JSON handling and enterprise feature set trail behind PostgreSQL.

---

## 4. ORM (Object-Relational Mapping)

**Selected Technology:** Prisma

- **Purpose:** To handle all database interactions, schema migrations, and type-safe data access from the backend layer.
- **Why it was selected:** Prisma provides end-to-end type safety directly from the database schema to the TypeScript codebase. Its schema declarative syntax accelerates development and simplifies database migrations, a critical velocity multiplier for a small team.
- **Alternatives considered:** TypeORM, Sequelize, Raw SQL.
- **Trade-offs:** Prisma abstracting complex SQL queries can lead to suboptimal query generation in highly specific edge cases, requiring raw SQL escape hatches.
- **Why alternatives were rejected:** TypeORM requires significant boilerplate and decorator pollution in domain entities, violating clean architecture principles. Raw SQL lacks type safety and slows down rapid iteration.

---

## 5. Analysis Engine

**Selected Technology:** Node.js Child Processes + `simple-git`

- **Purpose:** To ingest, traverse, and parse local Git repository histories to compute engineering metrics.
- **Why it was selected:** `simple-git` acts as a lightweight, promise-based wrapper around the native Git executable. It avoids the complexities of compiling native bindings while still operating at high speed.
- **Alternatives considered:** `nodegit` (libgit2 bindings), Python scripts, Native Go Binary.
- **Trade-offs:** Relies on the host system having the Git executable installed rather than bundling a Git implementation internally.
- **Why alternatives were rejected:** `nodegit` is notorious for compilation failures across different OS architectures, which severely impacts self-hosted enterprise deployments (e.g., Docker alpine vs debian). A Native Go binary violates the single-language stack constraint for the MVP.

---

## 6. AI Provider Layer

**Selected Technology:** Custom Provider-Agnostic Abstraction over OpenAI-compatible SDKs

- **Purpose:** To communicate with external Large Language Models to generate the Repository Health Reports based on computed metrics.
- **Why it was selected:** Building a lightweight, custom abstraction layer ensures we adhere strictly to the "AI-Provider Agnostic" principle defined in the PRD. We will implement the standard OpenAI REST interface, which guarantees out-of-the-box compatibility with OpenAI, local Ollama instances, and virtually all enterprise self-hosted model wrappers.
- **Alternatives considered:** LangChain.js, LlamaIndex.
- **Trade-offs:** We must maintain our own prompt templating and retry logic rather than relying on a heavy framework.
- **Why alternatives were rejected:** LangChain introduces massive dependency bloat and highly opinionated abstractions that change rapidly. For GitPro's tightly scoped report generation, a heavy AI framework is unnecessary complexity.

---

## 7. Authentication

**Selected Technology:** GitHub OAuth (Library TBD)

- **Purpose:** To authenticate users and acquire the necessary OAuth tokens for read-only repository access.
- **Why it was selected:** The MVP requires GitHub OAuth to streamline onboarding and repository selection for engineering leaders.
- **Alternatives considered:** Various Node.js authentication libraries (Passport.js, NextAuth).
- **Trade-offs:** The exact implementation library remains deferred.
- **Why alternatives were rejected:** The implementation library will be selected during the Authentication Sprint after evaluating security, maintainability, ecosystem maturity, and long-term compatibility. We are keeping the architectural decision implementation-agnostic at this stage.

---

## 8. Containerization

**Selected Technology:** Docker & Docker Compose

- **Purpose:** To package the application, database, and dependencies into reproducible, isolated environments for local development, CI/CD, and production deployment.
- **Why it was selected:** Docker is the undisputed industry standard. Docker Compose allows the two-person team to spin up the entire stack (PostgreSQL, Express API, Python Service, Vite Frontend) with a single command (`docker-compose up`).
- **Alternatives considered:** Bare Metal / PM2, Podman.
- **Trade-offs:** Introduces container overhead and requires Docker engine management.
- **Why alternatives were rejected:** Bare metal deployments lead to "it works on my machine" issues and significantly complicate the onboarding process for enterprise customers attempting to self-host the platform.

---

## 9. Testing

**Selected Technology:** Jest (Backend) & Vitest (Frontend)

- **Purpose:** To execute unit, integration, and domain boundary tests.
- **Why it was selected:** Jest is the industry standard testing framework for Node.js, offering excellent support for Express backend testing. Vitest provides a Jest-compatible API but runs blazingly fast alongside Vite for the frontend.
- **Alternatives considered:** Mocha, Cypress.
- **Trade-offs:** Maintaining two separate test runners across the repository.
- **Why alternatives were rejected:** Mocha requires manual configuration of assertion libraries (Chai) and mocking tools (Sinon). Cypress is for End-to-End (E2E) UI testing, which is deferred from the MVP scope in favor of unit and API-level integration testing.

---

## 10. Documentation

**Selected Technology:** Swagger (OpenAPI 3.0) & Markdown

- **Purpose:** To document the system architecture (ADRs, FRS, PRD) and the REST API boundaries.
- **Why it was selected:** NestJS provides automated Swagger generation via TypeScript decorators, resulting in always-accurate API documentation with near-zero maintenance effort. Markdown keeps architectural decisions version-controlled alongside the code.
- **Alternatives considered:** Postman Collections, Notion, Confluence.
- **Trade-offs:** Markdown lacks advanced WYSIWYG editing features.
- **Why alternatives were rejected:** Notion and Confluence separate documentation from the codebase, leading to drift. Postman collections require manual syncing, whereas Swagger is generated deterministically from the code itself.

## Technology Compatibility Matrix

The following matrix maps the physical interaction pathways between the selected technologies.

| Source Component | Target Component | Protocol / Interface | Purpose |
|---|---|---|---|
| **React Dashboard** | **Express Backend** | HTTPS / REST | User interactions, fetching metrics, reports |
| **Express Backend** | **Python Analysis Service** | gRPC / REST | Triggering deep repository mining and static analysis |
| **Python Analysis Service** | **AI Provider Layer** | Internal API | Formatting structural metrics for LLM context |
| **AI Provider Layer** | **External AI Models** | HTTPS / REST | Passing structured metrics to generate AI insights |
| **Express Backend** | **PostgreSQL** | TCP / Prisma Client | Persisting users, repos, and analysis runs |
| **Docker Engine** | **All Components** | Container Network | Process isolation and port forwarding |
