# ADR-001 — Why GitPro Uses a Modular Monolith for Version 1

**Document Type:** Architecture Decision Record
**Date:** July 11, 2026
**Author:** Principal Software Architecture Team
**Classification:** Confidential — Engineering Contract

---

## 1. Status

**Accepted**

---

## 2. Context

GitPro is entering its Minimum Viable Product (MVP) development phase. To deliver on the core product vision, the architecture must align with the current operational constraints while preserving a pathway to the long-term enterprise vision.

- **Team Size:** The engineering team consists of two full-time developers.
- **MVP Timeline:** The delivery window for the MVP is strictly bounded to six calendar weeks.
- **Engineering Goals:** The primary goal is rapid validation of the core value proposition (AI-driven engineering intelligence) with the highest possible development velocity and the lowest possible operational overhead.
- **Product Maturity:** The product is in the V1 (validation) phase. The feature set is tightly scoped to single-repository analysis, persistent storage, and AI report generation.
- **Long-Term Vision:** The platform will eventually evolve to support multi-repository portfolio analysis, CI/CD integrations, and enterprise self-hosted environments requiring horizontal scalability.

---

## 3. Problem Statement

The choice of foundational application architecture is critical. It defines the deployment topology, local development workflow, and inter-component communication patterns. An incorrect architectural choice at this stage could either severely bottleneck development velocity (causing the team to miss the 6-week timeline) or create a rigid structure that prevents future enterprise scaling. 

We must select an architecture that maximizes immediate productivity for a small team while establishing clear boundaries that support future decomposition.

---

## 4. Considered Options

### Option A: Modular Monolith

A single deployable unit where all application components (authentication, repository ingestion, analysis engine, AI reporting, and dashboard serving) are logically separated by strict internal module boundaries but run within the same process.

- **Advantages:** Single deployment target, simplified local development setup, straightforward debugging, no network latency between components, and atomic deployments.
- **Disadvantages:** Risk of boundary erosion over time (tight coupling) if discipline is not maintained; requires vertical scaling for resource-intensive tasks (e.g., repository ingestion) rather than targeted horizontal scaling.
- **Operational Complexity:** Very Low. Requires only a standard web server container and a database.
- **Development Speed:** Very High. No distributed system challenges (eventual consistency, network failures, distributed tracing).
- **Testing Complexity:** Low. Integration tests can run entirely in-memory without orchestrating multiple services.
- **Deployment Complexity:** Very Low. A single artifact is promoted through environments.
- **Scalability:** Moderate. Scales vertically easily. Horizontal scaling requires replicating the entire application, which may be inefficient if only the analysis engine requires heavy compute.
- **Suitability for MVP:** Extremely High. Aligns perfectly with a two-person team and a six-week timeline.

### Option B: Microservices

A distributed architecture where domain capabilities (e.g., User Management, Repository Ingestion, Analysis Engine, AI Integration) are deployed as independent, network-isolated services communicating via APIs or event buses.

- **Advantages:** Independent deployability, targeted horizontal scaling, strict physical boundaries preventing coupling, and technology heterogeneity (services can use different runtimes).
- **Disadvantages:** Significant operational overhead, complex local development orchestration, network latency, distributed data management challenges, and high initial setup cost.
- **Operational Complexity:** Very High. Requires service discovery, distributed tracing, network policies, and complex CI/CD pipelines.
- **Development Speed:** Low initially. The overhead of defining API contracts, handling network partitions, and orchestrating local services reduces time spent building product features.
- **Testing Complexity:** High. Requires contract testing and complex end-to-end environment orchestration.
- **Deployment Complexity:** High. Requires careful orchestration of service versions and database schemas.
- **Scalability:** Very High. Individual components can scale horizontally based on specific load profiles.
- **Suitability for MVP:** Very Low. The cognitive load and infrastructure overhead exceed the capacity of a two-person team within a six-week window.

---

## 5. Decision

**GitPro V1 will use a Modular Monolith architecture.** 

All components will be compiled and deployed as a single application artifact. Internal module boundaries will be strictly enforced through domain-driven design principles to ensure that logical separation exists, even though physical separation does not.

---

## 6. Decision Rationale

The decision to adopt a Modular Monolith is driven primarily by team capacity, time-to-market constraints, and the nature of the MVP scope:

1. **Alignment with Constraints:** A two-person team cannot afford to spend the first two weeks of a six-week timeline configuring service meshes, API gateways, and distributed tracing. The Modular Monolith allows the team to begin delivering business logic on day one.
2. **Reduced Cognitive Load:** In a distributed system, developers must constantly think about network failures, retry logic, and eventual consistency. A single-process architecture eliminates these concerns, allowing the team to focus entirely on the domain complexity of engineering metrics and AI integration.
3. **Sufficient Scalability for V1:** The MVP is scoped to single-repository analysis initiated by human users. The load profile of V1 does not require the massive horizontal scalability that microservices provide. Vertical scaling of a single process is more than sufficient for the validation phase.
4. **Self-Hosting Compatibility:** GitPro's long-term vision includes enterprise self-hosting. A single-container application is drastically easier for enterprise IT teams to evaluate and deploy on-premise compared to a complex microservice topology.
5. **Preparedness for Decomposition:** By enforcing strict logical boundaries (the "Modular" aspect), the architecture protects the long-term vision. The application is a monolith by deployment, but modular by design, ensuring that components can be extracted cleanly when scale demands it.

---

## 7. Consequences

### Positive Consequences
- **Maximum Velocity:** The team can iterate rapidly across the entire stack without crossing repository or network boundaries.
- **Simplified Operations:** Continuous integration, deployment, and local environment setup are trivial.
- **Ease of Debugging:** A single process means stack traces are complete and straightforward to follow.

### Negative Consequences
- **Resource Contention:** Heavy repository analysis tasks will share compute resources with the web dashboard, potentially impacting UI responsiveness under load.
- **Deployment Coupling:** A minor change to the dashboard requires redeploying the entire analysis engine.

### Technical Debt Introduced
- We are intentionally deferring the implementation of dedicated background processing queues for repository ingestion. In V1, ingestion will run within the web request lifecycle or a simple in-memory thread pool.

### Migration Strategy
- To mitigate boundary erosion, we will enforce strict architectural linting rules. The Web layer may depend on the Analysis layer, but the Analysis layer must never depend on the Web layer. This one-way dependency graph guarantees that extraction remains possible.

---

## 8. Future Evolution

The architecture is designed to evolve gracefully as the product matures, user base grows, and team size expands. The anticipated evolution path is as follows:

1. **Modular Monolith (Current):** Single process, single database. All logic runs together.
2. **↓**
3. **Background Workers (V1.5):** The Monolith is split into a "Web" process and a "Worker" process. They share the same codebase but execute different roles, allowing repository analysis to be scaled independently of web traffic via a message queue.
4. **↓**
5. **Dedicated Analysis Service (V2):** The analysis engine is physically extracted into its own repository and service. This allows the compute-heavy repository ingestion to be optimized independently of the dashboard.
6. **↓**
7. **Dedicated AI Service (V3):** The AI orchestration and provider abstraction layer is separated to manage complex prompt chains, rate limiting, and caching independently.
8. **↓**
9. **Enterprise Architecture (V4+):** A fully distributed topology capable of multi-repository portfolio aggregation, supporting dynamic scaling across Kubernetes clusters for large-scale enterprise deployments.

---

## 9. Decision Summary

**Executive Summary:** 
To deliver the GitPro MVP within six weeks using a two-developer team, we have selected a **Modular Monolith** architecture. This approach maximizes immediate development velocity, minimizes operational complexity, and simplifies the initial enterprise self-hosted experience. We explicitly rejected a Microservices architecture because its distributed systems overhead would jeopardize the MVP timeline. To protect the product's long-term enterprise vision, strict internal boundaries will be enforced, ensuring the application can smoothly evolve into a distributed system (beginning with dedicated background workers) as scale and complexity demand.
